## Challenges & Growth Areas

### Railway/GCP Platform Surprises (appeared in 3+ stories)
- Outbound SMTP blocked by GCP → pivoted to Resend (Story 1.2)
- `@tailwindcss/oxide` native Linux bindings failing → required custom Dockerfile (Story 1.1)
- `DATABASE_URL` internal URL not resolving from local machine → switched to public proxy URL (Story 1.2)
- Next.js `output: standalone` doesn't auto-include `.next/static` or `public/` → manual Dockerfile copy (Story 1.2)

### Library Version Gotchas (appeared in 3+ stories)
- pg-boss v12 breaking changes vs old v7/v8 tutorials — `createQueue()` required, array handlers, `expireInSeconds`, named export (Story 1.6)
- next-auth v5 renamed providers, changed import paths (Story 1.2)
- Firebase CDN compat script version must match installed npm package (Story 1.5)

### Technical Knowledge Gap
- When platform surprises require deep internals knowledge (Railway networking, GCP constraints, ESM edge cases), the burden falls on the agent to bridge that gap proactively — not on Todd to figure it out mid-story

---

## Key Insights

1. **Railway/GCP platform surprises are predictable** — they follow a pattern. Pre-loading known gotchas into dev notes before hitting them is the right mitigation.
2. **Code review catches real issues every story** — the process works, don't skip or rush it.
3. **Precision in instructions is the product** — exact import paths, exact CLI commands, exact env var names, explicit HALT sections. This is the right approach for Todd's workflow and should be maintained.
4. **Manual-only testing is insufficient for business logic** — infrastructure stories can get by; feature stories with real business rules cannot.
